\chapter{方法设计}

为解决大语言模型在运维故障诊断中面临的领域知识不足、数据难组织和推理不规范等问题，本文从知识检索、双源证据融合和结构化推理三个方面进行方法设计，并通过原型系统实现和实验进行验证。本章首先阐述方法设计目标与技术路线，随后依次介绍知识检索方法、双源证据融合方法、结构化推理框架和原型实现，详细说明各模块的设计思路与实现细节。

为避免术语混用，本文统一约定如下：``方法''指完整技术方案，``模块''指可独立实现的功能单元，``框架''指用于组织推理流程的步骤结构。后文均按此口径表述。

\section{方法设计目标}

本文方法的设计目标包括以下四个方面。

第一，\textbf{知识检索目标}：针对通用大模型缺乏特定系统领域知识的问题，构建面向运维场景的知识库，通过检索增强机制为模型提供标准操作流程、历史故障案例等背景信息。知识库需要具备语义检索能力，能够根据问题上下文召回相关文档，弥补模型在特定领域知识上的不足。

第二，\textbf{数据融合目标}：针对日志和监控指标格式不同且难以直接用于推理的问题，设计双源证据融合方法，将异构数据统一组织为结构化分析上下文。两类数据的格式和语义各不相同，如何设计有效的数据融合策略是双源数据处理需要考虑的问题。

第三，\textbf{诊断推理目标}：针对根因分析任务中自由生成结果不规范、可追溯性不足的问题，设计基于结构化提示的推理方法，引导模型完成故障摘要生成、原因推断与修复建议输出。诊断过程需要可追溯，每个结论都应有相应的证据支撑。

第四，\textbf{验证目标}：通过原型系统实现和对照实验，量化各模块对诊断效果的贡献，验证方法在典型故障场景下的可行性。

在本文架构中，原型系统作为方法验证载体，采用四层结构实现知识检索、数据融合与推理分析三个核心方法模块。三个核心方法模块的作用如下：

（1）\textbf{知识检索模块}：通过向量检索从运维知识库中召回与当前故障场景相关的背景知识，为诊断提供领域先验。

（2）\textbf{数据融合模块}：将日志摘要、监控指标异常和检索知识统一组织为结构化分析上下文，为推理提供现场证据。

（3）\textbf{推理分析模块}：通过结构化提示引导大语言模型按照规范流程完成故障分析，保证输出的一致性和可追溯性。

原型系统的四层结构为：

（1）\textbf{用户交互层}：基于Streamlit框架构建，提供友好的Web界面。用户可以通过自然语言向系统描述故障现象，系统以结构化形式返回诊断结果，包括故障摘要、根因分析、修复建议和置信度评分。

（2）\textbf{服务接口层}：基于FastAPI框架实现RESTful API接口，包括诊断分析接口、知识检索接口、实验评估接口等。该层负责接收用户请求，协调下游服务并返回结果。

（3）\textbf{核心逻辑层}：包含三个核心模块——知识检索模块、数据融合模块和推理分析模块。知识检索模块负责从向量数据库中召回相关文档；数据融合模块负责将双源实时数据统一组织为分析上下文；推理分析模块负责引导大语言模型完成故障分析。

（4）\textbf{数据存储层}：包括ChromaDB向量数据库（存储运维知识）、MySQL关系数据库（存储用户对话历史）和文件系统（存储日志和指标数据）。

\section{面向运维文档的知识检索方法}

运维知识库的质量直接影响知识检索效果。本文提出一种面向运维文档结构的知识切分与检索方法，解决通用大模型缺乏特定系统领域知识的问题，使系统能够针对用户提出的诊断性问题，检索出与当前场景相关的背景知识和历史经验，为后续推理提供依据。

本文的知识检索方法与常规RAG的主要差异在于：（1）采用面向运维文档结构的层级切分策略而非固定长度切分；（2）引入服务名和故障类型元数据过滤条件；（3）检索结果作为证据服务于后续分析。

\subsection{知识库内容体系}

知识库涵盖以下四类文档：

\textbf{SOP文档}：标准操作流程文档，包含CPU故障、内存故障、数据库故障、网络故障等常见场景的标准排查步骤和修复方案，适合支撑标准化排障步骤召回。

\textbf{历史故障案例}：记录真实发生的故障事件及其处理过程，适合支撑相似场景类比。

\textbf{系统架构文档}：描述系统各组件的功能、依赖关系和部署方式，适合支撑依赖关系理解。

\textbf{告警响应手册}：定义各类告警的响应策略和处理优先级，适合支撑优先级判断和响应建议。

\subsection{知识向量化处理}

知识库构建过程包括以下关键技术环节：

\textbf{文档预处理}：对原始文档进行格式清理，去除HTML标签、特殊符号和冗余空白。对于Markdown格式的文档，解析其层级结构以保留标题和段落的语义关系。

\textbf{面向运维文档结构的层级切分策略}：采用层级递归切分算法，按文档层级结构进行递归切分。对于SOP文档，以"步骤"为最小切分单元；对于案例文档，以"诊断阶段"为切分边界。与固定长度切分相比，基于Markdown层级的切分方式能够更好保留SOP步骤与案例阶段的语义完整性。切分时保留每个片段的元信息（文档类型、服务名、故障类型等），便于后续的元数据过滤检索。

\textbf{向量嵌入表示}：使用sentence-transformers预训练模型（all-MiniLM-L6-v2）将文本片段转换为384维语义向量。该模型在语义相似度任务上表现较好，且推理效率较高，适合实时检索场景。嵌入时将元数据一并编码，支持在检索时按服务类型、故障类型等维度进行过滤。

\textbf{向量存储索引}：将向量与原始文本、元数据一同存入ChromaDB向量数据库。ChromaDB采用HNSW（Hierarchical Navigable Small World）索引算法，能够在保证检索精度的同时实现毫秒级的Top-K查询。

\subsection{语义检索策略}

知识检索时，系统采用混合检索策略以平衡召回率和精确度：

\textbf{语义向量检索}：根据用户问题生成查询向量，在向量数据库中执行余弦相似度搜索。该方法能够捕捉问题与文档的语义相似性，即使问题表述与文档用词不同也能命中。

\textbf{元数据过滤条件}：在向量检索结果基础上，根据问题中提及的服务名称和故障类型进行过滤。运维场景中同一类故障在不同服务上差异较大，仅靠语义相似度容易召回无关知识，因此引入服务名和故障类型过滤条件以提高召回相关性。

\textbf{相关性重排序}：对初步检索结果进行相关性评分，综合考虑向量相似度、关键词匹配度等因素，对结果进行加权排序，优先返回最相关的Top-K知识片段。

本文的知识检索作为分析前的知识增强步骤，为故障分析提供受场景筛选的背景证据。

\subsection{知识检索的效果分析}

为验证知识检索模块对诊断效果的贡献，本文设计了RAG-Only基线实验（详见第四章）。实验结果表明，仅使用知识检索增强即可将诊断准确率从29\%提升至43\%，说明领域知识能够直接提升诊断的专业性。

\section{双源证据融合方法}

在故障诊断场景中，单一数据源通常难以完整反映系统状态。仅依赖日志容易缺失趋势信息，仅依赖指标又难以定位具体事件。针对这一问题，本文提出双源证据融合方法，将日志摘要与异常指标统一组织为结构化分析上下文，并与检索模块输出联合使用，为后续故障分析提供数据基础。

该模块的设计思路是：先明确“证据分散”这一问题，再给出“时间窗+TraceID+异常指标”的融合方案，最后在第四章通过对照实验验证该模块带来的准确率提升（43\%至71\%）。

日志处理的目标不是保留全部原始内容，而是尽可能在压缩上下文长度的同时保留对故障分析最关键的错误模式与时间线信息。将数值指标转换为自然语言描述的目的，是降低大语言模型直接理解原始时序数值的难度。

\subsection{日志数据处理}

日志数据是故障诊断的重要依据，但原始日志具有体量大、噪声多的特点。本文设计了三级日志处理流水线，将海量的原始日志转化为结构化的摘要信息：

\textbf{第一级：基于时间窗与TraceID的日志关联}。在分布式系统中，单一服务的日志不足以反映故障全貌。系统首先以故障发生时间为基准，提取滑动时间窗（如故障前后5分钟）内的日志。系统提取异常请求的链路追踪标识（TraceID），通过该TraceID将分散在不同微服务组件中的日志串联起来，形成统一的上下文，以排除同一时间段内无关请求的干扰。

\textbf{第二级：日志分类}。将过滤后的日志按级别分类，重点关注ERROR和WARNING级别日志。通过正则表达式匹配常见错误模式（如Connection timeout、OutOfMemory、NullPointerException等），提取关键错误信息。

\textbf{第三级：日志摘要}。对错误日志进行聚类和统计，生成结构化的日志摘要。摘要包含以下信息：（1）错误类型分布——统计各类错误的出现频次；（2）错误时间线——按时间顺序排列关键错误事件；（3）错误上下文——保留错误前后的关键日志行，便于理解错误发生的前因后果。

\subsection{监控指标处理}

监控指标提供了系统运行状态的定量描述。本文设计的指标处理流程包括：

\textbf{异常指标识别}。按预设阈值识别异常指标。阈值可根据不同服务动态调整，常见的异常指标包括：CPU使用率（阈值90\%）、内存使用率（阈值85\%）、磁盘使用率（阈值95\%）、响应延迟（阈值500ms）、错误率（阈值1\%）等。

\textbf{指标异常描述}。将数值型指标异常转换为文本描述。例如，"CPU使用率从30\%飙升至95\%"转换为"CPU使用率异常升高，近5分钟内增长65个百分点"。这种文本化处理有助于大语言模型理解指标变化。

\textbf{指标关联分析}。分析不同指标间的关联关系。例如，CPU升高往往伴随响应延迟增加，内存泄漏通常表现为内存使用率持续上升。通过关联分析，可以识别故障的主要矛盾和次要表现。

\subsection{严重程度评估}

数据融合模块综合考虑以下因素评估故障严重程度：

（1）\textbf{异常指标数量}：异常指标越多，表明故障影响范围越广。本文采用加权计数法，CPU和内存异常权重为2，其余指标权重为1。

（2）\textbf{错误日志密度}：单位时间内ERROR日志的出现频次。频次越高，表明故障越活跃。

（3）\textbf{指标偏离程度}：异常指标相对于阈值的偏离程度。例如，CPU使用率达99\%比达91\%更为严重。

严重程度分为四级：LOW（低）、MEDIUM（中）、HIGH（高）、CRITICAL（严重）。评估结果作为元信息传递给分析模块，帮助判断故障的紧迫程度。

\section{基于结构化提示的分析推理框架}

为提升分析输出的一致性和规范性，本文设计基于结构化提示的推理框架，引导大语言模型按预定义的分析步骤完成故障分析。

\subsection{结构化提示设计原则}

为解决自由生成结果不规范、证据与结论脱节、输出格式不统一等问题，本文归纳出结构化提示设计的三条核心原则：

\textbf{证据驱动原则}：所有结论必须基于提供的证据。通过要求模型在输出分析前先列出支撑证据和反对证据，确保推理过程有据可依。

\textbf{分步推理原则}：引导模型按照规范的逻辑框架展开分析，而非直接输出结论。通过要求模型先进行现象总结、证据分析，再进行原因推断和修复建议，确保推理过程可追溯。

\textbf{结构化输出原则}：要求模型按JSON格式输出，便于后续解析和结果比较。标准化输出格式使得不同场景下的诊断结果可以横向对比。

本文设计的分析提示包含以下组成部分：

\textbf{角色设定与行为约束}。定义模型为资深运维工程师，明确其分析原则。通过角色设定与行为约束，将模型输出限制在运维故障分析场景内。

（1）\textbf{现象总结}——简洁描述当前异常表现，包括受影响的指标和服务；

（2）\textbf{证据分析}——从指标、日志和知识库中提取支持分析的关键信息；

（3）\textbf{原因推断}——列出1至3个可能的故障原因，对每个原因给出：置信度（0-1）、支撑证据、反对证据、验证方法；

（4）\textbf{修复建议}——针对最可能的原因给出可操作的修复步骤；

（5）\textbf{数据缺口}——识别现有证据中的不足，提出需要补充的信息。

\textbf{输出格式约束}。要求模型按JSON格式输出，便于后续解析和结果比较：

\begin{verbatim}
{
    "incident_summary": "一句话故障描述",
    "suspected_root_causes": [
        {
            "cause": "根因描述",
            "confidence": 0.85,
            "supporting_evidence": ["证据1", "证据2"],
            "contradicting_evidence": []
        }
    ],
    "reasoning_chain": ["推理步骤1", "推理步骤2"],
    "verification_steps": ["验证步骤1", "验证步骤2"],
    "repair_suggestions": ["建议1", "建议2"],
    "need_more_data": ["缺失数据1"]
}
\end{verbatim}

上述结构化提示的设计参考了LogPrompt和SuperLog的提示工程方法，结合运维故障诊断的任务特点进行了优化。

\subsection{后备规则分析机制}

大语言模型在运维场景中仍可能调用失败或输出异常，为保证方法完整性，本文引入规则后备分析机制。该后备机制的目的不是替代大语言模型推理，而是在模型不可用或输出异常时保证方法流程的可用性。

规则映射表定义了常见指标异常与可能原因的对应关系：

\begin{itemize}
\item CPU异常 + 高负载 $\rightarrow$ 可能的原因：CPU密集型任务、死循环、资源竞争；
\item 内存异常 + 持续增长 $\rightarrow$ 可能的原因：内存泄漏、缓存膨胀、对象未释放；
\item 数据库连接异常 $\rightarrow$ 可能的原因：连接池配置不当、慢查询阻塞、连接泄露；
\item 网络延迟异常 $\rightarrow$ 可能的原因：网络抖动、DNS解析问题、服务端过载。
\end{itemize}

后备规则分析的输出格式简化，但仍包含故障原因和修复建议，便于用户参考。

\section{原型系统实现}

原型系统的实现目标是为知识检索、数据融合和分析推理三个模块提供统一验证环境。原型系统采用Python语言开发，技术栈包括Streamlit、FastAPI、ChromaDB和OpenAI API。

原型系统提供四个人机交互入口：（1）故障诊断界面：用户输入故障描述，系统返回结构化诊断结果；（2）知识查询界面：用户可直接查询运维知识库；（3）监控展示界面：展示实时监控指标和告警信息；（4）实验评估界面：支持批量运行测试场景，查看评估指标和对比结果。

后端主要模块包括：（1）诊断服务：接收故障描述，协调知识检索、数据融合和分析推理，返回诊断结果；（2）知识检索服务：封装向量检索逻辑，支持按服务名、故障类型等维度过滤；（3）实验评估服务：支持批量运行预设测试场景，计算评估指标。

端到端的诊断Pipeline执行流程为：接收用户问题后，首先调用知识检索服务获取相关文档；然后调用日志和指标服务获取实时故障信息；接着通过数据融合模块将双源实时信息统一组织为分析上下文；最后调用分析推理模块生成诊断结果。
