\chapter{方法设计与研究验证原型}

本章围绕本文的三个研究问题展开方法设计。整体思路是将日志与监控指标协同分析作为主线方法，将知识增强和结构化推理作为两个辅助模块，并通过一个最小化的研究验证原型承载实验。与偏产品化的系统设计不同，本文更关注每个模块在诊断链路中的职责是否清楚、变量是否便于控制以及输出是否便于评估。

\section{设计目标与总体框架}

本文方法的设计目标可概括为三个层次。

第一，\textbf{核心诊断目标}：针对单一数据源难以稳定支撑根因定位的问题，构建日志与监控指标协同分析流程，以统一现场证据为后续诊断提供基础。

第二，\textbf{辅助增益目标}：针对知识依赖型故障场景，引入检索增强补充背景知识；针对输出格式不稳定问题，引入结构化推理提高结果可审查性。

第三，\textbf{验证目标}：通过统一原型环境和可重复的评分规则，验证核心方法与辅助模块的作用边界，而不是仅展示系统界面或功能数量。

\section{日志与监控指标协同分析方法}

\subsection{日志处理策略}

日志处理的目标不是保留全部原始日志，而是在有限上下文内尽量保留与故障判断最相关的信息。结合日志分析研究中对模板稳定性和噪声控制的关注~\cite{du2017deeplog,zhang2019logrobust,liu2024logprompt}，本文将日志处理分为三个步骤。

\textbf{第一，时间窗过滤。}以故障发生时段为中心，提取前后固定时间窗内的日志。这样做的目的是缩小上下文范围，避免将过多无关日志输入后续模块。

\textbf{第二，请求链路关联。}对于能够识别TraceID或请求标识的样例，将同一请求链上的日志进行归并，以减轻分布式场景中“日志多而分散”的问题。若场景中缺少显式链路标识，则退化为按服务与时间顺序组织日志。

\textbf{第三，错误摘要生成。}统计ERROR和WARNING级别日志，提取高频错误模式、关键报错语句和时间线特征，生成供模型直接理解的日志摘要文本。该摘要不追求复刻全部原始日志，而是强调故障定位所需的错误类型、发生顺序和上下文线索。

\subsection{监控指标处理策略}

监控指标处理的目标，是将原本偏数值和时序的数据转换为对诊断更友好的证据表达。具体包括以下三个步骤。

\textbf{第一，异常识别。}针对CPU使用率、内存使用率、磁盘占用率、请求延迟、错误率、数据库连接数等指标设置经验阈值，并结合变化趋势判断其是否异常。本文使用简单阈值与趋势结合的策略，而不引入复杂时序模型，主要考虑到本科论文的实验重点在于证据协同而不是单独的异常检测算法比较。

\textbf{第二，指标文本化。}将“95\%”“持续上升”“5分钟内增长60个百分点”等数值变化转写为自然语言描述，以降低模型直接处理原始序列的难度。例如，将“P95延迟由320ms升至1450ms”表达为“接口延迟在短时间内明显上升，已超过设定阈值”。

\textbf{第三，指标关联。}对不同指标间的共现关系进行简化归纳。例如，CPU升高伴随延迟上升更可能指向计算密集或资源竞争；连接数逼近上限且错误率增加更可能指向连接池耗尽或阻塞。这一步不直接给出根因，而是服务于后续上下文组织。

\subsection{统一分析上下文构建}

在完成日志摘要和指标处理后，系统将双源证据组织为统一分析上下文。该上下文至少包含六项内容：用户问题、服务标识、时间范围、日志摘要、指标摘要和异常指标列表。当RAG模块返回知识片段时，知识结果以辅助字段附加在上下文中；严重程度评估则作为辅助元信息传递给推理模块。

统一分析上下文的作用有两点。第一，它为后续推理模块提供结构化、长度受控且来源清晰的证据集合。第二，它使不同实验组之间的输入差异保持可比，例如Metric-Only组只保留指标字段，Log-Only组只保留日志字段，Log+Metric组同时保留两类核心证据。

\section{辅助知识增强方法}

\subsection{知识库内容组织}

结合RAG研究中“知识检索应服务于具体任务”的观点~\cite{gao2023ragsurvey,jiang2023active}，本文仅选取与运维诊断直接相关的文档进入知识库，包括三类内容：标准操作流程文档、历史故障案例和与关键服务相关的简要架构说明。这些文档既能提供排障步骤，也能为慢SQL、连接池耗尽等场景补充运维经验背景。

\subsection{层级切分与元数据过滤}

在知识切分上，本文采用面向运维文档结构的层级切分策略，而不是固定长度切分。具体而言，优先按Markdown标题层级、步骤编号和案例阶段划分文档片段，使每个片段尽量保持语义完整。这一做法的原因在于，运维文档中的“步骤”“现象”“处理建议”通常本身就具有天然结构，如果简单按字数切分，容易打断关键语义单元。

在检索阶段，除了语义相似度外，还增加服务名称、故障类型等元数据过滤条件。这样做的目的是减少“语义相似但服务无关”的召回结果，提高知识片段与当前场景的匹配程度。该思路与运维RAG研究中“场景限定优于无约束召回”的观点一致~\cite{zhang2024rag4itops,isaza2024ragincident}。

\subsection{知识增强的使用边界}

本文明确将RAG定位为辅助模块，而不是诊断主框架。只有当场景具有明显的知识依赖特征时，检索结果才进入统一分析上下文；若故障可仅依靠日志和指标判断，则RAG不应主导结论。这样的设计有利于后续实验中观察知识增强对不同故障类型的作用差异。

\section{辅助结构化推理方法}

\subsection{结构化提示设计原则}

结合提示工程和运维诊断任务特征~\cite{liu2023prompt,wei2022cot,wang2023planandsolve}，本文将结构化提示的设计原则概括为三点。

\textbf{证据驱动原则。}所有根因假设都必须显式对应支持证据，避免先下结论再补理由。

\textbf{分步分析原则。}模型需要先进行现象总结，再执行证据分析、原因推断、验证建议和修复建议，避免将不同推理步骤混杂在一段自然语言中。

\textbf{结构化输出原则。}输出统一为固定字段集合，以便后续解析、评分和案例比较。

\subsection{结构化输出字段}

本文要求模型输出以下核心字段：故障摘要、根因假设列表、支持证据、反对证据、验证步骤、修复建议和数据缺口。字段设计并不追求复杂，而是围绕实验评估需要确定最小集合。其参考格式如下：

\begin{verbatim}
{
  "incident_summary": "一句话概括异常现象",
  "suspected_root_causes": [
    {
      "cause": "根因描述",
      "confidence": 0.82,
      "supporting_evidence": ["证据1", "证据2"],
      "contradicting_evidence": []
    }
  ],
  "verification_steps": ["步骤1", "步骤2"],
  "repair_suggestions": ["建议1", "建议2"],
  "need_more_data": ["需要补充的信息"]
}
\end{verbatim}

\subsection{后备规则机制}

考虑到大语言模型在实验环境中可能出现调用失败、字段缺失或输出不可解析等情况，本文保留一个简化的规则后备机制。该机制仅依据已识别的异常指标和高频错误模式返回粗粒度分析结果，用于保证实验流程可以完整执行。需要说明的是，该后备机制不作为本文的研究创新点，也不承担主要诊断任务，只作为实验鲁棒性补充。

\section{研究验证原型设计}

本文原型系统的定位是\textbf{研究验证原型}，而非完整产品系统。其作用是为方法验证提供统一环境，而不是追求交互功能的完备性。原型仅保留四类能力：场景管理、日志与指标处理、知识检索与结构化诊断、批量实验评估。

原型采用Python技术栈实现，后端使用FastAPI组织服务接口，前端使用Streamlit提供最小化展示界面，知识片段存储于向量数据库中，实验样例和评分逻辑以本地文件形式管理。图~\ref{fig:pipeline-flow}给出了原型中的端到端执行流程。

\section{本章小结}

本章给出了本文的方法设计。首先明确了“双源协同为主线、知识增强与结构化推理为辅助”的总体框架；随后介绍了日志处理、指标处理和统一分析上下文构建等核心方法；在此基础上说明了RAG与结构化推理两个辅助模块的设计原则；最后介绍了研究验证原型的最小实现边界。上述内容为第四章的实验设置和结果分析提供了方法依据。
