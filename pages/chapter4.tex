\chapter{实验与结果分析}

为了验证LLM-Ops Copilot原型系统的可行性与有效性，本章设计了典型的故障测试场景，通过对照实验评估各模块对系统效果的贡献，并从多个维度分析实验结果。实验旨在回答以下问题：（1）各技术模块对诊断准确率的贡献分别是多少？（2）完整方案相比基线方法有多大的提升？（3）系统在典型故障场景下的表现如何？

\section{实验环境}

原型系统采用Python语言开发~\cite{zhang2024rag4itops}，主要依赖包包括Streamlit、FastAPI、ChromaDB、LangChain和OpenAI SDK。大语言模型调用采用OpenAI API接口~\cite{hou2023llm4se}，使用GPT-4模型（版本gpt-4-0613）进行推理。向量数据库使用ChromaDB~\cite{zhang2024rag4itops}，文本嵌入使用sentence-transformers预训练模型（all-MiniLM-L6-v2）。

系统部署在通用计算环境下，配置为8核CPU、32GB内存。日志数据存储在本地文件系统，监控指标通过模拟数据生成器产生~\cite{pham2024rcacausal}。实验环境的具体配置如表~\ref{tab:experiment-env}所示。

\section{评估指标}

本文采用以下指标评估系统效果~\cite{liu2023logprompt}：

（1）\textbf{根因定位准确率}：判断系统输出的根因是否命中真实故障原因~\cite{roy2024agents}。采用关键词交集法——若系统输出根因与真实根因的关键词交集数量$\ge$2，则视为正确。

（2）\textbf{Top-3准确率}：若真实根因出现在系统输出的前3个候选根因中，则视为命中~\cite{zhang2024rag4itops}。

（3）\textbf{响应时间}：从用户提交问题到系统返回结果的总耗时~\cite{zhang2024rag4itops}。

（4）\textbf{证据覆盖率}：分析结果中提及的证据来源（如日志、指标、知识库）占总证据来源的比例~\cite{liu2023logprompt}。

（5）\textbf{修复建议有效率}：由专家评估修复建议的可操作性和正确性~\cite{zhang2024rag4itops}。该指标衡量系统生成的修复建议在实际运维中的可用程度。

\section{对照实验设计}

为验证各模块对系统效果的贡献，本文设计了四组对照实验：

（1）\textbf{LLM-Only}：仅依赖大语言模型，不使用RAG、不融合实时数据~\cite{zhang2024rag4itops}。

（2）\textbf{RAG-Only}：使用RAG知识检索增强，但不融合实时日志和监控指标。该方案用于验证知识库对诊断效果的独立贡献。

（3）\textbf{RAG+Realtime}：同时使用RAG和实时数据融合，但不采用结构化Prompt~\cite{liu2023logprompt}。该方案用于验证实时数据融合的独立作用。

（4）\textbf{Full-Method}：完整方案，使用RAG、实时数据融合和结构化Prompt~\cite{zhang2024rag4itops,liu2023logprompt,roy2024agents}。该方案综合了本文提出的所有技术要素。

实验结果表明：

（1）\textbf{LLM-Only方案效果有限}。仅依赖大模型直接回答时~\cite{zhang2024rag4itops}，系统在专业性方面存在明显不足，平均准确率仅为43\%。这是因为通用大模型缺乏运维领域知识，容易产生泛化回答，而非针对具体故障场景的专业诊断~\cite{zhang2024rag4itops}。

（2）\textbf{RAG知识增强显著提升效果}。引入RAG后~\cite{zhang2024rag4itops}，准确率提升至57\%，提升了14个百分点。知识库提供了运维领域的专业背景信息，使模型能够根据具体场景给出更准确的判断。

（3）\textbf{实时数据融合进一步改善}。融合实时日志和监控指标后~\cite{pham2024rcacausal}，准确率提升至71\%，相比RAG-Only方案又提升了14个百分点。实时数据提供了故障的现场信息，使模型能够结合实际情况进行分析。

（4）\textbf{结构化Prompt发挥关键作用}。完整方案的准确率达到86\%~\cite{liu2023logprompt,roy2024agents}，相比RAG+Realtime方案提升了15个百分点。结构化Prompt约束了模型的推理过程，引导模型按照规范的逻辑框架展开分析，提升了输出的一致性和准确性。

\subsection{典型案例分析}

为深入理解各模块的作用机制，本文以服务雪崩场景（S6）为例，展示系统在有无数据融合时的诊断输出差异~\cite{pham2024rcacausal}。

\textbf{场景描述}：某电商平台在促销活动期间，订单服务出现大量超时告警，响应延迟从正常值50ms飙升至2000ms以上~\cite{pham2024rcacausal}。

\textbf{LLM-Only方案输出}：
\begin{verbatim}
根据您的描述，订单服务响应延迟增加可能有以下原因：
1. 网络问题
2. 数据库负载过高
3. 服务器资源不足

建议检查网络连接、数据库状态和服务器资源使用情况。
\end{verbatim}

上述回答存在以下问题：（1）回答过于泛化，没有针对性~\cite{zhang2024rag4itops}；（2）缺乏具体证据支撑，论证过程不透明；（3）修复建议不具体，难以直接操作。

\textbf{Full-Method方案输出}：
\begin{verbatim}
{
  "summary": "订单服务响应延迟从50ms飙升至2000ms，
              伴随商品服务ERROR日志激增",
  "analysis": {
    "evidence_summary": "商品服务15:23-15:25期间ERROR日志
    从0激增至47条，关键词：Connection refused(23)，
    Timeout(18)，Fallback triggered(6)",
    "root_causes": [
      {
        "cause": "下游商品服务故障导致订单服务连锁
                 超时（服务雪崩）",
        "confidence": 0.92,
        "supporting_evidence": [
          "商品服务ERROR日志在15:23开始激增",
          "订单服务延迟峰值与商品服务ERROR激增时间吻合",
          "知识库相似案例：'促销高峰下游服务超时导致雪崩'"
        ],
        "contradicting_evidence": [
          "订单服务本身CPU/内存指标正常"
        ],
        "verification_steps": [
          "检查商品服务健康状态",
          "查看商品服务详细错误日志",
          "确认是否存在超时配置过短问题"
        ]
      }
    ]
  },
  "primary_fix": {
    "action": "启用熔断机制，隔离故障服务",
    "steps": [
      "1. 临时调高订单服务对商品服务的超时阈值",
      "2. 启用Fallback机制，返回缓存数据",
      "3. 联系商品服务团队确认故障原因",
      "4. 故障恢复后逐步恢复流量"
    ]
  },
  "confidence": 0.92
}
\end{verbatim}

对比可见，Full-Method方案的输出具备以下优势：（1）引用了具体的日志证据和时间线，增强了结论的可信度~\cite{liu2023logprompt}；（2）给出了高置信度的根因假设和置信度评分，便于运维人员快速判断；（3）提供了可操作的修复步骤~\cite{roy2024agents}；（4）明确指出了数据缺口（商品服务本身CPU/内存正常），体现了分析的严谨性。

该案例验证了证据融合和结构化Prompt对于提升诊断质量的关键作用~\cite{pham2024rcacausal,liu2023logprompt}。

\subsection{分模块效果分析}

为量化各模块对系统效果的贡献~\cite{ji2024superlog}，本文进行了消融分析，结果如下。

（1）\textbf{知识检索模块的作用}。移除RAG模块后，系统在S5（慢SQL查询）场景中出现明显回答泛化~\cite{zhang2024rag4itops}。例如，对于"查询响应缓慢"的问题，系统给出了"增加服务器资源"的建议，而非针对性的SQL优化建议~\cite{zhang2024rag4itops}。消融实验表明，知识检索模块对专业性提升有显著作用，平均准确率贡献约14\%。

（2）\textbf{证据融合模块的作用}。移除证据融合模块后~\cite{pham2024rcacausal}，系统在S6（服务雪崩）场景中无法准确识别故障传播路径。模型虽然知道"服务超时"，但无法将延迟增加与下游服务的ERROR日志关联起来~\cite{pham2024rcacausal}。消融实验表明，证据融合模块对复杂故障分析至关重要，Top-3准确率贡献约14\%。

（3）\textbf{结构化Prompt的作用}。移除结构化Prompt后~\cite{liu2023logprompt}，模型的输出格式变得不统一，有时是段落文本，有时是列表，有时甚至包含Markdown格式问题。此外，模型输出的置信度评分也变得不可靠~\cite{liu2023logprompt}。消融实验表明，结构化Prompt对输出一致性贡献最大。

\section{实验误差与局限}

尽管实验结果验证了所提方法的基本可行性~\cite{zhang2024rag4itops}，但本次实验与原型系统仍存在一定的误差与局限。

第一，测试场景与真实生产环境存在差异。本次实验在受控微服务下通过注入脚本模拟故障产生，虽然按照真实的Prometheus指标格式和日志范式进行了还原，但真实环境中的噪声与并发状态更为复杂，可能导致实际准确率波动。

第二，大模型存在潜在的“幻觉”误差。在实验中发现，即使应用了结构化Prompt进行约束，大语言模型在极少数情况下仍可能根据错误的日志组合编造不存在的系统依赖。后续需引入交叉验证机制来消除此类误差。

第三，评估指标较为单一~\cite{liu2023logprompt}。当前采用关键词交集法评估根因定位准确率，该方法对同义词和表述差异较为敏感，可能导致实际有效但表达不同的答案被误判为错误。

第四，响应时间受限于大模型生成延迟。实验中完整方案的响应时间达到34.8秒，这主要是因为GPT-4级别大模型生成长文本时的生成延迟所致。虽然目前已处于可接受范围内，但仍需探索轻量化模型或流式输出机制来改善体验。

此外，本文的对照实验数量相对有限，测试场景为7个。虽然这些场景覆盖了常见的故障类型，但在统计显著性方面仍有提升空间。未来可考虑增加更多测试场景，并进行统计显著性检验，以增强实验结论的可靠性。

\section{本章小结}

本章通过7个典型故障场景验证了LLM-Ops Copilot原型系统的可行性~\cite{zhang2024rag4itops}。实验结果表明~\cite{liu2023logprompt,roy2024agents}：

（1）仅依赖大语言模型时准确率仅为43\%，表明通用模型在专业运维场景下存在明显不足，容易产生泛化回答；

（2）引入RAG知识增强后准确率提升至57\%，说明领域知识对诊断效果有显著提升，知识检索能够为模型提供专业的领域背景信息；

（3）进一步融合实时数据后准确率提升至71\%，验证了多源数据融合的重要性，实时故障信息对于准确诊断至关重要；

（4）完整方案的准确率达到86\%，Top-3准确率达到100\%，证据覆盖率达到78\%，充分验证了所提方法的有效性，结构化Prompt在提升诊断准确性方面发挥了关键作用。

本章同时客观分析了系统当前存在的不足，如测试规模有限、评估方法单一和缺乏横向对比等，为后续改进指明了方向。实验结果为本文提出的基于大模型的软件运维方法提供了实证支持，也为智能运维领域的研究提供了参考。
