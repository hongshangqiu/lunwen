# 论文修改总方案｜研究型重构与文献替换清单

<aside>
🧭

本页用于汇总论文终稿前的修改方案。整体目标是将论文稳定收束为一篇**方法导向、原型验证、结论克制**的研究型本科论文。

</aside>

## 一、修改总目标

本文修改应围绕以下四个目标展开：

- **研究定位更清楚**：突出“问题—方法—实验—结论”的研究逻辑，弱化“系统功能介绍”式写法。
- **表达尺度更稳妥**：结论、创新点和实验表述应符合本科论文可支撑范围，避免夸大。
- **实验部分更严谨**：保证表格、百分比、案例说明和指标定义前后一致。
- **形式层面更规范**：清除残留草稿、异常引用、未完成内容和低级格式错误。

## 二、总体修改原则

### 2.1 研究叙事原则

全文应尽量从“为解决什么问题，设计了什么方法，如何验证其有效性”展开，而不应继续沿用“做了哪些模块、实现了哪些页面”的工程叙事。

### 2.2 本科论文尺度原则

本文应定位为：

- 面向软件运维故障诊断任务的方法研究；
- 通过原型系统完成实现承载；
- 通过受控实验验证方法可行性。

因此，正文中可以使用“验证了方法的可行性”“表明该方法在所设场景下具有较好效果”等表述，但不宜使用过强的结论，如“充分证明”“显著领先”“可直接推广到真实生产环境”等。

### 2.3 形式与内容统一原则

文中所有核心概念、字段名、表格数据、实验结果和文献引用应保持统一。凡是不能与全文主线形成直接支撑关系的内容，应压缩、删除或改写。

## 三、优先修改项

### 3.1 第一优先级：先修硬伤

以下问题应优先处理，且必须在整体润色前完成。

#### （1）统一实验数据

需逐项核对以下内容是否一致：

- 中文摘要中的准确率、Top-3 准确率、证据覆盖率；
- 第 4.5.1 节中的柱状图数值；
- 表 4.3、表 4.4 中的统计结果；
- 第 4.5.3 节消融分析中的模块贡献表述；
- 第 4.7 节本章小结中的结果复述。

若采用 7 个场景逐项统计，则所有准确率必须能由场景结果直接推导；若采用多次实验取平均值，则需在正文中明确说明统计方式，不能再以单一勾叉表替代平均结果。

#### （2）补全英文摘要

英文摘要必须补成完整版本，内容结构应与中文摘要一致，包括：

- 问题背景；
- 方法组成；
- 实验设计；
- 结果结论；
- 关键词。

同时检查是否存在断句异常、乱码、连字符断裂或句子未写完的情况。

#### （3）清理异常引用

全文删除残留的草稿式引用标记或异常混合引用，如未清理的作者缩写、BibTeX 键值、拼接错误的编号等。正文统一使用标准编号引用格式。

#### （4）统一方法定义与案例展示格式

第 3 章给出的结构化输出格式、第 4 章案例展示中的 JSON 字段、附录代码中的关键字段命名应统一为同一套表达。不能出现“方法定义一套字段，案例展示另一套字段”的情况。

### 3.2 第二优先级：再修研究表达

在完成硬伤修正后，再统一处理下列表达问题：

- 收束“创新点”“贡献”“显著提升”等过强措辞；
- 压缩与论文主线关系不大的原型系统描述；
- 增强实验结果的机制解释，而非仅汇报数字变化。

## 四、逐章修改方案

### 4.1 第一章：绪论

本章应继续强化问题导向和研究缺口，保持研究型叙事。

修改重点如下：

- 1.1 中保留现实背景，但避免泛泛而谈，重点落在故障诊断中的知识不足、数据异构和推理不稳定三个问题上；
- 1.2 中保持“日志分析—RAG—RCA—研究缺口”的结构，不再扩展无关背景；
- 1.3 中继续用“研究内容”而非“系统功能”表达；
- 1.4 中强调文献调研法、方法设计法、原型验证法和对照实验法；
- 1.5 中说明后续各章如何围绕研究主线展开。

建议：本章不再继续增加新内容，重点是使措辞更简洁、更稳妥。

### 4.2 第二章：相关技术基础

本章应保留必要技术背景，但避免教材化展开。

修改重点如下：

- 2.1 只保留与 AIOps 演进和本文问题直接相关的内容；
- 2.2 保留上下文学习、分步推理和结构化输出三项与本文直接相关的大模型能力；
- 2.3 重点写清楚“为什么运维场景需要 RAG”，而不是泛泛介绍 RAG 流程；
- 2.4 重点说明日志与指标在 RCA 中的互补性；
- 2.5 将根因分析表述为“证据归纳—根因假设—验证与建议”的过程；
- 2.6 保留技术适配性分析，使第 2 章与第 3 章自然衔接。

建议：若本章篇幅偏长，可优先压缩通识性描述，保留与后文方法直接对应的论述。

### 4.3 第三章：方法设计与原型实现

本章是全文核心，应继续突出三个方法模块，压缩“实现展示”色彩。

#### （1）3.1—3.2 的修改重点

- 明确区分“研究核心三部分”与“原型落地四层结构”；
- 避免前后出现“三层架构”与“四层结构”表述冲突；
- 强调原型系统只是方法验证载体，不是论文主体。

#### （2）3.3 的修改重点

- 保留面向运维文档结构的切分与检索策略；
- 强化“为什么按 Markdown 层级切分优于固定长度切分”；
- 明确元数据过滤在运维场景中的必要性；
- 将知识检索定位为后续 RCA 的背景证据来源，而非独立问答模块。

#### （3）3.4 的修改重点

- 保持证据融合写法，可适当增加“统一分析上下文”的解释；
- 明确日志摘要和指标文本化的目的，是为后续大模型推理降低上下文噪声；
- 严重程度评估保持为辅助信息，不要写成独立创新点。

#### （4）3.5 的修改重点

- 强调结构化 Prompt 的三条原则：证据驱动、分步推理、结构化输出；
- 统一输出字段格式，与第 4 章案例展示一致；
- 将后备规则机制写成鲁棒性补充，而非主方法替代。

#### （5）3.6 的修改重点

- 压缩界面、接口和交互入口的细节；
- 保留原型系统如何支撑实验即可；
- 避免使用过多产品说明式表述。

### 4.4 第四章：实验与结果分析

本章是决定论文可信度的关键章节，应优先加强一致性和严谨性。

#### （1）4.1—4.4 的修改重点

- 明确实验环境属于受控原型环境；
- 明确日志、指标和知识库数据的来源与构造方式；
- 说明 7 个测试场景的覆盖逻辑；
- 说明 4 组对照实验分别用于验证什么。

#### （2）4.5 的修改重点

- 在 4.5.1 中突出“为什么提升”，而不是只写“提升了多少”；
- 在 4.5.2 中强化案例对比逻辑：输入差异、证据差异、推理差异、输出差异；
- 在 4.5.3 中继续保持“消融分析与模块贡献讨论”的研究型写法。

#### （3）4.6 的修改重点

本节应更明确地承认以下边界：

- 实验基于模拟环境，不能等同于真实生产验证；
- 测试场景数量有限，统计显著性不足；
- 评价指标采用近似方法，存在表达偏差风险；
- 当前工作主要验证辅助诊断能力，尚未验证自动修复能力。

建议：本章结论要稳，不宜写成“已证明方法可直接应用于真实工业环境”。

### 4.5 第五章：总结与展望

本章应收束结论与创新点，控制在本科论文可支撑范围内。

#### （1）5.1 研究工作总结

建议按“问题—方法—实验—结论”四步总结，不再强调系统界面、平台功能和工程实现细节。

#### （2）5.2 主要创新点

创新点建议稳定保留为三项：

- 面向运维文档结构的知识切分与检索策略；
- 面向 RCA 的日志与指标证据融合机制；
- 面向故障诊断任务的结构化 Prompt 推理框架。

表述上应采用“提出了……策略/机制/框架”的尺度，不宜拔高为理论突破或通用工业方案。

#### （3）5.3 未来研究展望

保留知识库自动更新、多模态数据融合、复杂传播链建模、真实场景验证等方向，但应明确这些属于后续工作，而非本文已完成内容。

## 五、统一修改规则

### 5.1 术语统一

需统一以下术语的写法：

- 大语言模型 / LLM
- 检索增强生成 / RAG
- 根因分析 / RCA
- 多源证据融合
- 结构化 Prompt
- 原型系统
- 受控原型环境

保证关键词大小写、连字符和专业术语表达统一。

### 5.2 表达统一

全文建议优先使用以下表达：

- “本文针对……问题提出……”
- “为解决……问题，设计……”
- “实验结果表明……”
- “说明该方法在所设场景下具有较好效果……”

尽量减少以下表达：

- “本文设计并实现了完整平台”
- “系统功能如下”
- “充分证明”
- “显著领先”
- “可直接落地生产环境”

### 5.3 图表与附录统一

- 图表名称、编号和正文引用保持一致；
- 图中英文缩写首次出现时给出全称；
- 附录代码字段与正文方法描述保持一致；
- 若某图表仅起展示作用、不能支持核心论点，可适当压缩或删除。

## 六、建议执行顺序

### 第一步：先修硬伤

1. 统一实验数据；
2. 补全文摘要；
3. 清理异常引用；
4. 统一输出结构。

### 第二步：再修章节表达

1. 收紧第 3 章中的系统实现表述；
2. 增强第 4 章中的结果解释；
3. 收束第 5 章中的结论与创新点。

### 第三步：最后做通读清稿

1. 检查术语统一；
2. 检查图表编号和交叉引用；
3. 检查参考文献格式；
4. 检查摘要、中英文关键词、目录、附录和致谢的完整性。

## 七、最终目标

修改完成后，论文应达到以下状态：

- 结构上符合“问题—方法—实验—结论”的研究逻辑；
- 内容上突出方法设计与实验验证，弱化工程展示；
- 表达上符合研究型本科论文尺度；
- 实验上做到数据一致、边界清楚；
- 形式上避免低级错误，具备稳定交稿条件。
