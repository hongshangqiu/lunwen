## 1. 系统概述

### 1.1 项目背景与定位

**项目名称**：LLM-Ops Copilot（基于大模型的智能运维故障诊断与自愈系统）

**核心定位**：面向软件系统的智能运维辅助工具，通过集成实时监控、日志分析和运维知识库，利用大语言模型实现故障的自动化诊断与根因定位。

**研究问题**：
- 如何有效利用大模型的推理能力进行故障根因分析？
- 如何将结构化的运维知识与实时监控数据融合？
- 如何设计 Prompt 使 LLM 输出可信赖、可验证的诊断结果？

### 1.2 核心功能

| 功能模块 | 描述 | 论文支撑点 |
|---------|------|-----------|
| 故障注入 | 模拟 CPU 飙升、数据库连接池耗尽、网络延迟等真实故障 | 实验验证基础 |
| 多源数据融合 | 融合指标、日志、知识库三类数据源 | 3.2 节方法描述 |
| RAG 知识增强 | 从运维知识库检索相关 SOP 手册 | 3.1 节方法描述 |
| 智能 RCA | 基于 LLM 的结构化根因分析 | 3.3 节方法描述 |
| 自动化自愈 | AI 生成修复脚本并支持一键执行 | 扩展应用场景 |
| 对比实验评估 | 四组对照实验量化方案效果 | 4.3 节实验验证 |

### 1.3 创新点总结

1. **多源数据融合架构**：提出将监控指标、应用日志、运维知识库统一融合的方法
2. **结构化 Prompt 设计**：设计包含证据分析、置信度评估、数据缺口识别的结构化 Prompt
3. **混合推理策略**：结合 LLM 推理与规则匹配的后备机制，提高系统鲁棒性
4. **对比实验框架**：设计四组对照实验量化不同方案贡献度

### 3.2 模块 2：多源数据融合 (`app/core/evidence_builder.py`)

#### 3.2.1 功能描述

证据融合器是系统的核心枢纽，负责：
1. 接收来自不同数据源的数据
2. 提取异常指标
3. 评估故障严重程度
4. 构建统一的分析上下文

#### 3.2.2 异常指标阈值配置

```python
self.thresholds = {
    "cpu": 80.0,           # CPU 使用率阈值 (%)
    "memory": 85.0,        # 内存使用率阈值 (%)
    "latency": 1000.0,     # 延迟阈值 (毫秒)
    "error_rate": 5.0,     # 错误率阈值 (%)
    "connections": 90.0,   # 连接数阈值 (%)
}
```

#### 3.2.3 严重程度评估规则

| 评分因素 | 规则 |
|---------|------|
| 异常指标数量 | ≥3: +3分, ≥2: +2分, ≥1: +1分 |
| 严重错误日志 | 存在: +3分 |
| 错误日志数量 | >5: +2分, >0: +1分 |
| 警告日志数量 | >10: +1分 |
| 指标峰值 | >95%: +1分, >90%: +0.5分 |

**严重程度映射**：
- 5分及以上 → CRITICAL
- 3-5分 → HIGH
- 1-3分 → MEDIUM
- 0分 → LOW

### 3.3 模块 3：RCA 推理引擎 (`app/services/rca_engine.py`)

#### 3.3.1 功能描述

基于 LLM + 结构化 Prompt 的根因分析引擎，支持：
- 结构化 JSON 输出
- 多根因假设并列
- 置信度评分
- 后备规则分析

#### 3.3.2 RCA 结果数据结构

```python
@dataclass
class RCAResult:
    incident_summary: str          # 故障摘要
    suspected_root_causes: List[RootCause]  # 根因假设列表
    reasoning_chain: List[str]    # 推理步骤链
    verification_steps: List[str] # 验证步骤
    repair_suggestions: List[str] # 修复建议
    need_more_data: List[str]     # 数据缺口
    raw_response: str             # 原始响应
    parse_success: bool           # 解析是否成功
    parse_error: str              # 解析错误信息
```

#### 3.3.3 后备规则分析

当 LLM 分析失败时，系统自动切换到基于规则的简单分析：

| 异常指标 | 推断根因 |
|---------|---------|
| CPU 相关 | CPU 使用率异常升高，可能存在 CPU 密集型任务或攻击行为 |
| Memory 相关 | 内存使用率异常，可能存在内存泄漏或内存配置不足 |
| Connection 相关 | 数据库/服务连接异常，可能存在连接池耗尽或网络问题 |
| Latency 相关 | 系统延迟升高，可能存在性能瓶颈或网络问题 |
| Error 相关 | 系统错误率升高，可能存在代码异常或依赖服务故障 |

### 3.4 模块 4：Prompt 模板设计 (`app/core/prompts.py`)

#### 3.4.1 系统级提示词

```python
SYSTEM_PROMPT = """你是一名资深 SRE（网站可靠性工程师），专精于软件故障诊断。

你的专业背景：
- 10 年以上分布式系统运维经验
- 精通 Linux 内核、性能调优、容器技术
- 熟悉常见故障模式：CPU 飙升、内存泄漏、网络延迟、数据库连接池耗尽等

你的分析遵循以下原则：
1. 证据驱动：所有结论必须基于提供的指标、日志或知识库内容
2. 结构化输出：严格按照指定格式返回结果
3. 置信度评估：每个根因假设需给出置信度评分
4. 可操作建议：修复建议必须具体可执行
5. 诚实透明：证据不足时明确说明，不编造信息"""
```

#### 3.4.2 RCA 主 Prompt 结构

```
## 故障分析任务

### 输入上下文
{context_json}

### 推理步骤（请按顺序思考）

**Step 1: 现象总结**
- 简洁描述系统当前的异常表现（1-2句话）

**Step 2: 证据分析**
- 从指标中识别哪些指标出现异常？变化幅度如何？
- 从日志中提取哪些关键错误信息？
- 检索到的知识库内容提供了哪些背景信息？

**Step 3: 根因假设**
- 基于证据，列出 1-3 个可能的根因
- 每个根因必须包含：原因描述、置信度、支持证据、反对证据

**Step 4: 验证计划**
- 列出验证每个根因假设需要做的具体检查

**Step 5: 修复建议**
- 针对最可能的根因，给出可操作的修复步骤

**Step 6: 数据缺口**
- 如果当前证据不足以支撑确定结论，明确说明还需要哪些数据

### 输出格式（严格遵循 JSON）
{
    "incident_summary": "一句话故障描述",
    "suspected_root_causes": [...],
    "reasoning_chain": [...],
    "verification_steps": [...],
    "repair_suggestions": [...],
    "need_more_data": [...]
}
```

```python
def retrieve(query, service=None, fault_type=None, top_k=3):

    RAG 检索算法
    
    Input: query (str), service (str), fault_type (str), top_k (int)

    1. 构建过滤器
```

## 5. 数据结构定义

### 5.1 AnalysisContext（分析上下文）

```python
@dataclass
class AnalysisContext:
    """统一分析上下文"""
    question: str                           # 用户问题
    service: str                            # 服务名称
    time_range: str                         # 时间范围
    retrieved_knowledge: List[Dict]        # 知识库检索结果
    metrics_summary: Dict                   # 指标摘要
    log_summary: Dict                      # 日志摘要
    anomaly_indicators: List[str]          # 异常指标列表
    severity: str                           # 严重程度: low/medium/high/critical
    raw_metrics: Optional[Dict] = None      # 原始指标数据
    raw_logs: Optional[Dict] = None         # 原始日志数据
```

### 5.2 RootCause（根因假设）

```python
@dataclass
class RootCause:
    """根因假设"""
    cause: str                              # 根因描述
    confidence: float                      # 置信度 (0.0-1.0)
    supporting_evidence: List[str]         # 支持证据
    contradicting_evidence: List[str]      # 反对证据
```

```python
@dataclass
class RCAResult:

    repair_suggestions: List[str]          # 修复建议
    need_more_data: List[str]              # 数据缺口
    raw_response: str = ""                  # 原始响应
    parse_success: bool = True              # 解析是否成功
```

## 6. 实验设计

### 6.1 实验目标

通过四组对照实验，量化评估：
1. RAG 知识增强的贡献
2. 实时数据融合的贡献
3. 结构化 Prompt 的贡献

### 6.2 实验分组

### 7.3 根因匹配算法

```python
def is_root_cause_correct(ground_truth, predicted):
    """
    判断预测根因是否正确
    
    算法：
    1. 将 ground_truth 和 predicted 转为小写
    2. 提取关键词（正则匹配 \w+）
    3. 计算关键词交集
    4. 如果交集 >= 2，认为正确
    """
    ground_keywords = set(re.findall(r'\w+', ground_truth.lower()))
    predicted_keywords = set(re.findall(r'\w+', predicted.lower()))
    overlap = ground_keywords & predicted_keywords
    return len(overlap) >= 2
```
